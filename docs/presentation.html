Snake in C
— from the metal up —

███████ ●
A terminal game written with malloc, <string.h>, <math.h>, fopen — all hand-rolled.
Welcome. This is a classic Snake game, written from scratch in C. What makes it interesting is what we ~didn't~ use: no malloc, no <string.h>, no <math.h>, no fopen. We rolled all of it ourselves. By the end of this talk you'll understand every layer, from the 64 KB memory pool up to the BFS-driven auto-play AI.
premise

The challenge

Build a full-featured Snake game in C without the beginner-friendly crutches.

what's banned
  • malloc / free
  • <string.h> — strlen, strcpy, …
  • <math.h> — random, abs, clamp
  • fopen / fprintf for save files
what we built anyway
  • 3 modes · 3 difficulties
  • Persistent per-mode high scores
  • BFS-driven AI auto-play
  • 2-player local with race-fair input
  • Gradient snake that shimmers over time
Why self-impose these restrictions? To actually understand what those library functions do. Once you write malloc yourself, you'll never look at it the same way.

The bottom layer has zero dependencies. Each module is a public .h (what it offers) and a private .c (how it does it).

We have ten .c files. At the bottom: memory, math, string — they depend on nothing. Everything else builds upward. Clean separation of concerns, same as a Python package but explicit.
01 · build

The Makefile

Turns 10 .c files into ./snake, rebuilds only what changed.

CFLAGS = -Wall -Wextra -pedantic -std=c99 \
         -D_DEFAULT_SOURCE -I include -MMD -MP

$(TARGET): $(OBJS)
    $(CC) $(CFLAGS) -o $@ $^

$(BUILD_DIR)/%.o: $(SRC_DIR)/%.c | $(BUILD_DIR)
    $(CC) $(CFLAGS) -c -o $@ $<
-Wall -Wextra

Crank warnings. Catch bugs early.

-MMD -MP

Auto-track headers. Edit game.h → everything that includes it rebuilds.

-I include

Where to look for #include.

Make is dumb but effective. It sees which .c files are newer than their .o files, rebuilds just those, then re-links. The -MMD -MP flags are the hero: they auto-generate dependency files so header changes trigger correct rebuilds. We learned that the hard way.
02 · memory

A 64 KB malloc, by hand

Grab one big byte array. Track usage with a linked list of headers living inside the pool.

┌──────────────────────────────────────────────────────────┐
├────────────┬────────────┬────────────┬────────────┬──────┤
│ BlockHeader│  user data │ BlockHeader│  user data │ ...  │
│ size|free  │  (n bytes) │ size|free  │  (n bytes) │      │
│ next─┐     │            │ next─┐     │            │      │
└──────┼─────┴────────────┴──────┼─────┴────────────┴──────┘
       └──────────────────────────┘
alloc

Walk list, first-fit, split remainder, return pointer past the header.

free

Step back 16 B to find header, flip free=1, merge adjacent free neighbours.

64 KB is our entire memory budget. Every snake segment, every linked-list node, comes from this pool. Inside the pool we keep a linked list of bookkeeping headers — each says "I'm this big, I'm free or used, next is at X". Eighty lines of C.
03 · math

A PRNG from scratch

No rand(). We use a Linear Congruential Generator — the simplest pseudo-random algorithm in existence.

int math_rand(int lo, int hi) {
    g_seed = (g_seed * 1103515245 + 12345) & 0x7fffffff;
    int range = hi - lo + 1;
    return lo + (int)(g_seed % range);
}
the formula

next = (cur · a + c) mod m

a = 1,103,515,245  ·  c = 12,345  ·  m = 2³¹

seeded with

time(NULL) at startup.

Different second → different food positions.

Every call multiplies by a huge number, adds a small one, and chops to 31 bits. The output looks random but is fully deterministic given the seed. Same algorithm POSIX's rand uses. Perfect for a snake game; don't use it for passwords.
04 · string

int → string, the hard way

A string in C is just a sequence of bytes ending in \0. So we hand-rolled str_len, str_copy, and the key one: int_to_str.

// digits come out in REVERSE — then we flip
while (value > 0) {
    buf[i++] = (value % 10) + '0';   // last digit → ASCII
    value = value / 10;              // chop it off
}
// reverse in place
for (j = 0; j < i/2; j++) swap(buf[j], buf[i-1-j]);

423  →  "3","2","4"  →  reverse  →  "423" ✅
Why it matters: this is exactly how we serialise high scores to disk. No printf needed.

The "+ '0'" trick: '0' is ASCII 48, so digit + '0' gives the character. Extract digits with mod 10 and divide 10 — they come out in reverse — then swap-loop to fix. It's how formatted I/O works under the hood.
05 · screen

The terminal as a canvas

We draw by printing ANSI escape codes — magic byte sequences the terminal interprets as commands instead of text.

SequenceWhat it does
\x1b[2JClear entire screen
\x1b[5;10HMove cursor to row 5, col 10
\x1b[38;5;196mSet foreground to palette color 196 (red)
\x1b[?25lHide the cursor
\x1b[?1049hSwitch to alternate screen (!)

Alt-screen is why your shell history isn't trashed after you quit. And we flush stdout once per frame (64 KB buffer) → zero flicker.

That's all "graphics" is here: printf magic strings. Two tricks make it smooth — alternate screen buffer so we don't stomp the user's shell, and a 64 KB output buffer we flush once per frame so no partial frames ever render.
06 · keyboard

Raw mode — every key, instantly

Default terminals wait for Enter. We need each key the moment it's pressed.

raw.c_lflag &= ~(ICANON | ECHO);   // turn off line buffering + echo
raw.c_cc[VMIN]  = 0;
raw.c_cc[VTIME] = 0;
tcsetattr(STDIN_FILENO, TCSAFLUSH, &raw);
fcntl(STDIN_FILENO, F_SETFL, flags | O_NONBLOCK);
the arrow-key trick

Arrow keys arrive as a 3-byte escape sequence: ESC [ A = Up. We detect the ESC, read 2 more bytes, and map to custom codes 128–131 — outside ASCII so they never collide with WASD. Critical for 2-player mode.

Two changes: raw mode via termios gives us each key as a byte, non-blocking via fcntl so read returns immediately if there's nothing. Arrow keys are 3 bytes — we detect the ESC, peek two more, and return code 128 for up, 129 for down, etc. Outside ASCII so WASD and arrows never confuse each other.
07 · score

Persistence, syscall-level

No fopen. We use raw open / read / write / close from <unistd.h>.

// SAVE
fd = open(path, O_WRONLY | O_CREAT | O_TRUNC, 0644);
int_to_str(score, buf, 32);
write(fd, buf, str_len(buf));
close(fd);

// LOAD
fd = open(path, O_RDONLY);
n  = read(fd, buf, 31);
close(fd);
buf[n] = '\0';
return str_to_int(buf);

One file per mode: .snake_highscore_easy, .snake_highscore_ai, … The contents are literally the digits. Separate records for Easy / Normal / Hard / AI / 2-Player.

A file descriptor is just an int — the OS's handle to your open file. We open, write the ASCII digits directly, close. The pair int_to_str/str_to_int does the serialisation. No formatting library needed.
08 · game

The snake is a linked list

       ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
 head→ │ 10,5 │ →  │  9,5 │ →  │  8,5 │ →  │  7,5 │ → NULL
       └──────┘    └──────┘    └──────┘    └──────┘
           ▲                                    ▲
       head ptr                            tail ptr  (cached!)

Each tick:

  1. Prepend a new head segment (mem_alloc)
  2. Did we eat food?  — keep the tail → snake grows
  3. Otherwise?  — mem_free the old tail → length unchanged

O(1) at both ends thanks to the cached tail pointer.

Head and tail pointers let us do constant-time operations at both ends. The snake is literally a linked list in the 64 KB pool. Growing is "don't free the tail"; shrinking is free it. That's the whole data model.
08 · game

Collision rules (every tick)

For each snake's candidate new-head cell, check against…

1 · self

own body, minus tail if not growing (tail will move out of the way)

2 · cross

other snake's body (multi mode), same tail rule

3 · head-to-head

same destination cell → mutual KO → draw

Walls wrap around (snake re-enters the opposite side). Speed ramps down by 4 ms per food eaten, capped at 40 ms.

The subtle bit is the tail exclusion: if the snake isn't growing this tick, its tail is going to move out of the way — so we don't count it as a collision. That lets the snake chase its own tail at distance 1 safely, which is the classic "snake rule".

10 · ai

AI mode: BFS pathfinding

Every tick, the AI runs a breadth-first search from head → food.
The first move along the shortest path is the action.

  before BFS                   after BFS (ripple)
  . . . . .                    3 . . . .
  . . . F .                    2 1 . . .
  . . H . .        →           1 H 1 . .     numbers = distance
  . . . . .                    2 1 2 . .
  . . . . .                    3 . . . .

Like dropping a stone in water. The first cell of the reconstructed path tells us which direction to move right now.

BFS is the textbook shortest-path algorithm on an unweighted grid. Drop a stone in water; ripples spread outward one step at a time. We stop as soon as the ripple hits the food cell — that's our shortest path.
10 · ai · implementation

One queue, one parent table

qx[qtail] = hx;  qy[qtail] = hy;  qtail++;
visited[hy][hx] = 1;

while (qhead < qtail) {
    cx = qx[qhead]; cy = qy[qhead]; qhead++;
    if (cx == fx && cy == fy) break;          // ← found food

    for (d = 0; d < 4; d++) {
        nx = wrap_x(cx + DX[d], board_w);
        ny = wrap_y(cy + DY[d], board_h);
        if (visited[ny][nx] || blocked(nx, ny)) continue;

        visited[ny][nx] = 1;
        parent[ny][nx]  = d;                  // ← remember the move
        qx[qtail] = nx; qy[qtail] = ny; qtail++;
    }
}
// walk back from food to head → first_dir is the answer

Static arrays (no allocation). Wraparound-aware. Fallback to "any safe direction" if no path exists. Runs in microseconds per tick.

Queue is two parallel arrays with head/tail indices — no dynamic allocation. We record the direction used to enter each cell in a parent table. When we reach the food, we walk backwards using parent[] and the first move of that walk is what we return.
11 · main

The game loop (per tick)

while (1) {
    if (resize_flag) redraw_everything();

    p1_pending = -1;  p2_pending = -1;
    while (keyPressed()) {                    // DRAIN all bytes
        int key = readKey();
        // route 'w/a/s/d' → p1_pending, arrows → p2_pending
    }

    if (mode == MODE_AI) p1_pending = ai_decide(game);

    game_set_player_dir(game, 1, p1_pending);
    game_set_player_dir(game, 2, p2_pending);
    game_update(game);
    game_render(game);

    // interruptible sleep (10 ms chunks, break on input)
    sleep_until_next_tick();
}
Every tick: drain all pending input bytes, let the AI decide if in AI mode, apply the pending directions, update the world, render, sleep until the next tick. The sleep is interruptible so we can react to key presses instantly instead of waiting the whole 200 ms.
11 · main · the subtle bug

Same-millisecond keypress race

Both players press at once. Both bytes arrive in the buffer in the same read(). Naive code reads one byte per tick → one player starves.

// FIX: per-player slots + drain ALL bytes
int p1_pending = -1, p2_pending = -1;
while (keyPressed()) {
    int k = readKey();
    switch (k) {
      case 'w': p1_pending = DIR_UP;   break;
      case 's': p1_pending = DIR_DOWN; break;
      // ...
      case KEY_UP:   p2_pending = DIR_UP;   break;
      case KEY_DOWN: p2_pending = DIR_DOWN; break;
      // ...
    }
}
// Both applied atomically at update time

Within one player, last press wins. Across players, both get their move every tick.

This was the trickiest bug. If you only read one byte per tick, when both players press together one byte sits in the buffer for a tick — that player feels laggy. The fix is two things: drain every byte per tick, AND route each into a per-player slot so P1's latest doesn't overwrite P2's. The distinct arrow key codes from keyboard.c make the routing unambiguous.

retrospective

Lessons I'd pin to the wall

  • "A string is just bytes ending in zero."
    Internalising this collapses most of C's "weirdness" for a Python/JS dev.
  • Stable input with a pending-dir buffer.
    Decouples input frequency from tick rate. Essential for smooth gameplay.
  • Always enable -MMD -MP.
    Stale .o files after header changes will ruin your Saturday.
  • Low-level I/O is simpler than formatted I/O.
    open/read/write is 4 syscalls. fopen/fprintf is thousands of lines of machinery.
If you only take four things from this talk: strings are just bytes ending in zero, decouple input from tick rate with a buffer, always track headers in your Makefile, and the lowest-level I/O is often the easiest to reason about.
